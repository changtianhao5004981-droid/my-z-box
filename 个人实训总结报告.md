## 一、项目概述与个人贡献

本次居家实训，我独立完成了全栈 Web 项目 **Z-Box** 的设计、开发、部署与文档编写。项目基于 Node.js + Express 5 构建后端，前端采用原生 HTML / CSS / JavaScript，数据库使用 MongoDB Atlas，AI 能力通过 OpenRouter 接入多种大模型，并部署至 Render 平台供线上访问。

在项目中的具体贡献如下：

- **架构设计**：确定「Express 后端代理 + 静态前端 + 第三方 API 聚合」的整体方案，将 6 个功能页面与 8 个后端接口统一在一个服务中。
- **后端开发**：独立完成 `server.js` 中全部路由，包括 AI 流式对话（SSE）、新闻列表与详情、必应壁纸、短视频、资源搜索，以及 MongoDB 用户注册 / 登录接口。
- **前端开发**：实现 AI 对话页（Markdown 渲染、Cookie 聊天记录）、新闻分页加载、网盘搜索分页、词典 / 壁纸 / 短视频轮播页，以及全站可拖拽的登录注册悬浮组件。
- **工程化与部署**：配置 dotenv 环境变量管理、Git 版本控制、Render 云端部署，并编写 README、API 文档与 Prompt 开发日志。

## 二、遇到的最大困难及解决过程

### 困难一：第三方 API 返回结构不一致

项目集成了天行新闻、Qzqi 短视频、Bing 壁纸等多个第三方接口，文档描述与实际返回往往对不上。例如短视频接口文档写的是 `code: 0`，实测成功时却是 `code: 1`，视频链接嵌套在 `data.data.link` 中；新闻列表也不在顶层数组，而是藏在 `result.newslist` 里。

**解决过程：** 我先用 Postman 和浏览器 Network 面板抓取真实响应，把 JSON 结构记录下来，再借助 Cursor AI 编写数据清洗逻辑，在后端统一转换为前端需要的格式（如 `{ items: [...] }`、`{ status, link }`）。这个过程让我深刻认识到：**对接外部 API 时，「看文档」不如「看实际响应」可靠**，后端代理层的价值就在于屏蔽这些差异。

### 困难二：AI 流式对话的 SSE 实现

AI 对话是项目的核心功能，需要把 OpenRouter 的流式响应实时推送给前端。最初我尝试用普通 fetch 一次性获取完整回复，用户体验很差；改用 SSE 后，又遇到了事件类型解析（`answer` / `reasoning` / `done`）、换行符转义、连接中断处理等问题。

**解决过程：** 后端参考 OpenRouter 的 SSE 协议，逐行解析 `data:` 前缀的 JSON chunk，按类型推送不同 event；前端用 `EventSource` 监听，配合 marked.js 做增量 Markdown 渲染。调试时在服务端加了 `console.log` 输出上游原始数据，定位到 DeepSeek R1 会额外返回 `reasoning` 字段。最终实现了「边想边输出」的流式体验。

### 困难三：云端部署与环境变量管理

本地开发一切正常，部署到 Render 后却出现 MongoDB 连接失败、AI 接口 401 等问题。排查发现是环境变量未正确配置，以及 Render 免费实例冷启动导致首次访问超时。

**解决过程：** 将所有敏感信息（MongoDB URI、OpenRouter API Key）迁移到 Render 环境变量面板，本地用 `.env` + `.gitignore` 隔离；在 README 中注明冷启动等待时间，并录制本地启动视频作为兜底验证材料。这次经历让我理解了**「能在本地跑」和「能在公网跑」之间还有一层工程化工作**。

## 三、AI 辅助编程的心得体会

本次实训全程使用 Cursor AI 作为编程助手，我有以下几点体会：

**1. AI 适合「加速」而非「替代」思考**

AI 能在几分钟内生成 Express 骨架、SSE 代理模板、分页组件等 boilerplate 代码，大幅缩短重复劳动时间。但如果 Prompt 描述不清晰，AI 生成的代码往往与项目上下文不匹配。我逐渐学会在 Prompt 中附上**实际 API 响应 JSON、目标文件路径、期望返回格式**，这样 AI 的输出质量明显提高。AI 是加速器，但**需求分析和架构决策仍然需要开发者自己完成**。

**2. Prompt 日志是过程留痕的关键**

考核方案强调 Git 提交记录和 Prompt 日志作为「考勤表」。我在开发过程中养成了习惯：每完成一个功能模块，就在 `prompt_log.md` 中记录 Prompt、AI 输出摘要和对应文件。这不仅满足考核要求，也方便日后回顾「这段代码是怎么来的」。当 AI 给出错误方案时，日志帮助我快速定位问题 Prompt 并修正。

**3. Code Review 不能省略**

AI 生成的代码可能存在安全隐患（如密码明文存储、API Key 硬编码）。我在提交前用 Cursor 做了一次 Code Review，发现并修正了新闻接口 Key 写死在代码中的问题（后续应迁移至环境变量）。**AI 辅助编程的完整流程应该是：Prompt → 生成 → 理解 → 审查 → 测试 → 提交**，而不是复制粘贴了事。

**4. 从「写代码」到「管工程」的转变**

居家实训没有现场答辩，考核权重向代码质量、工程规范和文档倾斜。这让我意识到，一个合格的项目不仅要功能能跑，还要有清晰的目录结构、可复现的安装指南、完整的 API 文档、真实的 Git 历史和可追溯的 AI 使用记录。AI 工具把「写代码」的门槛降低了，但**「工程化」的要求反而更高了**。

## 四、收获与展望

通过本次实训，我完整走通了一个 Web 项目从 0 到 1 的全流程：需求拆解 → 技术选型 → 前后端开发 → 第三方 API 对接 → 数据库集成 → 云端部署 → 文档编写。相比以往只做前端或只做后端的练习，这次的项目更贴近真实开发场景。

不足之处也很明确：用户密码尚未加密存储，缺少单元测试和 CI/CD 自动化部署，错误监控和日志系统也比较简陋。如果有更多时间，我计划引入 bcrypt 密码哈希、Jest 接口测试，以及 GitHub Actions 自动部署，作为加分项继续完善。

总的来说，这次居家实训让我真正体验了 AI 辅助编程的工作方式——**它不是让开发者变懒，而是让开发者把精力集中在更有价值的架构设计、问题排查和工程规范上**。Z-Box 项目虽然功能并不复杂，但它是我独立完成、可线上访问、文档齐全的第一个全栈作品，我对这个结果感到满意，也从中建立了继续深入 Web 开发的信心。
